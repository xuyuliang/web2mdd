# 英语单词形变推演与联想系统·综合规划方案

## 一、系统核心定位与价值

本系统旨在解决英语学习中"形似词易混淆"和"词根词缀理性记忆"的痛点。它不是一个死记硬背的"查表"工具，而是一个**符号主义推演引擎**。

系统通过学习用户现有的 Anki 样本数据，内化构词法规律；当面对未来输入的全新、陌生单词时，能够像外科手术一样对其进行拆解，并利用模糊和衍生规则，"繁衍"出尽可能多的、真实的、按词频排序的关联词，帮助用户建立视觉和逻辑上的双重联想。

**设计原则**：纯算法驱动，硬件友好——此设计完全基于文本切片、字典映射和闭环 API 交互，运行时在毫秒级，普通电脑即可流畅运转。

---

## 二、系统核心架构：双轨驱动引擎

系统在处理单词时，内部并行运行两条核心轨道：

```
                    ┌─── [ 轨道 A：词根模糊推演轨道 ] ───> 剥离前后缀 -> 提取词根 -> 辅音骨架/元音模糊形变
                    │
[输入全新未知新词] ──┤
                    │
                    └─── [ 轨道 B：词缀变换平移轨道 ] ───> 保持词根 -> 变换/轮替前后缀 -> 派生新形态
```

### 轨道 A（结构切分与模糊推演）
- **主攻**：理性记忆（词根溯源）
- **流程**：剥离前后缀 → 拿到核心词根 → 允许词根进行元音交替或辅音骨架的押韵变幻
- **示例**：`thump → stump / trump`（保持 -ump 骨架，替换首辅音丛），`inveterate → veteran`（共享词根 veter）
- **核心锚点**：提取词根的**辅音骨架**（去除元音 a/e/i/o/u/y），作为后续模糊匹配的基准

### 轨道 B（形似词检索与变换）
- **主攻**：感性辨析（词族扩展）
- **流程**：在保持词根相对稳定的情况下，通过在两头大范围换头（前缀）或改尾（后缀），批量产生衍生词
- **示例**：`implicit ↔ explicit`（前缀 im-/ex- 互换）、`respective → respectively`（后缀 -ive → -ly 拓展）

## 三、核心算法流程（四阶段闭环）

### 阶段 1：知识沉淀阶段 —— 从现有数据中"学规律"

**输入源**：1000+ 条 Anki 历史数据（包含 `source_word`, `source_seg`, `related_word`, `related_seg` 等字段）以及一份基础的前后缀补充知识表。

**动态前沿知识挖掘（重点）**：
系统不能完全依赖死板的前后缀表。它需要读取训练集中的 `source_seg` 和 `related_seg`（如 `re.spec.tive.ly`），通过**点号（.）切分**，动态、自动地将高频出现在两端的片段识别并沉淀为新的前后缀。

**规律矩阵构建**：
1. **前缀-后缀转换矩阵**：自动识别并记录前后缀的替换关联（如看到 `implicit → explicit`，记录 `im-` 与 `ex-` 具备高频转换可能）
2. **辅音骨架库**：提取词根的辅音骨架（去除元音 aeiouy，如 `disparate` 提取出 `d-s-p-r-t`），作为后续模糊匹配的锚点
3. **变形规律频次统计**：统计样本中每种变形规律（如去掉 `-ly`、`im-` 换 `ex-`）的出现频次，为阶段4的权重评分提供依据

### 阶段 2：新词拆解阶段 —— 对未知新词"动手术"

当接收到一个完全不在表中的新单词时，系统进行以下拆解：

1. **最长匹配原则**：利用前后缀表和动态挖掘的前后缀库，由两头向中间最大化剥离已知的前缀和后缀
2. **剥离约束**：确保中间的"核心词根"留有合理的字母空间（不少于2-3个字母），锁定该新词的"核心形态特征"
3. **产物**：`[前缀?] + 核心词根 + [后缀?]` 的三段式结构

### 阶段 3：模糊推演阶段 —— 多路径"繁衍新词"

基于双轨架构，新词进入三条路径同时形变：

#### 🅰 路径 A：换头（前缀变换）
- **操作**：保持词根和后缀不变，将新词的原前缀替换为前缀表或规律库中同组、高频互换的其他前缀
- **示例**：`in.veter.ate` → 替换 `in-` 为 `re-`, `con-`, `per-` → `re.veter.ate?` 等

#### 🅱 路径 B：改尾（后缀变换）
- **操作**：保持前缀和词根不变，根据后缀表进行词性转换或尾部平移
- **示例**：`respect.ive` → 替换 `-ive` 为 `-ion`, `-ly`, `-ing` → `respect.ion?` 等

#### 🅲 路径 C：动内脏（词根元音 / 辅音骨架变幻）
- **操作1**：允许词根内部的元音字母（a/e/i/o/u）有限互换
- **操作2**：基于辅音骨架库的辅音演变（如 `th- → str-`, `th- → st-`, `squ- → squ-` 等押韵规律）
- **示例**：保持 `-ump` 骨架，尝试各种声母组合 → `thump, trump, stump, plump, clump, slump`

> 💡 **三条路径可以组合使用**：例如先执行路径C改变词根，再执行路径B添加新后缀，衍生出更丰富的候选词集合。

### 阶段 4：智能筛选与输出阶段 —— "自适应闭环机制"

这是文档3重点补充的闭环控制逻辑，也是系统质量的核心保障。推演出的所有候选词，必须经过以下漏斗式校验：

```
[路径A/B/C 产生大量候选词]
           │
           ▼
  ┌─ 调用单词存在性 API ──────(查不到)──────┐
  │        │(查到了)                         │
  │        ▼                                 │
  │  调用 COCA 词频 API                      ▼
  │        │                      触发[自适应微调]：
  │        ▼                      动态微调/替换元音字母
  │  合理性权重评分 ──────────── 再次查 API ──→ 循环直至
  │        │                      (有限次数内)    找到/放弃
  │        ▼
  │  分级排列输出
  │        │
  └────────┘
```

#### 4.1 单词存在性校验
所有衍生词先调用系统已做好的 `is_valid_word()` 或 `mdx_reader.lookup(word)` 判断单词是否真实存在。如果查不到 → 触发自适应变化（如元音微调/增删字母），变化后再次查询，直到捕捉到真实单词或达到重试上限。

#### 4.2 词频获取
将通过校验的真实单词送入 `mdx_reader.get_rank(word)` 捕获 COCA 词频排名数据。

#### 4.3 合理性权重评分（新增重点）
此为文档2补充的关键机制。评分基于两个维度：

| 评分维度 | 权重 | 说明 |
|---------|------|------|
| **词频** | 60% | 词频rank值小，则高评分 |
| **形似度分值** | 40% | 编辑距离 + 辅音骨架相似度综合评分 |

#### 4.4 分级输出（新增重点）
此为文档2提出的重要输出策略：
- **第一梯队（高概率相关词）**：完全符合真实英语构词规律、变形跨度较小的高评分词，排在前面优先展示
- **第二梯队（模糊猜测词）**：变形跨度较大、评分较低的衍生词，排在后面作为扩展参考

最终结果严格按照 **"评分权重 × 词频"** 从高到低降序排列展示，实现**"由准到深、由常用到生僻"**的完美输出效果。

## 四、与现有系统的对接

> ⚠️ **实况核对**：以下接口并非以孤立的"独立函数名"存在，而是嵌入在 `MDXReader` 词典读取器中的方法。总结文档中的 `check_word_exists_api()` 和 `get_word_frequency_api()` 是规划中的抽象描述，实际代码中对应的实现如下：

| 规划中的抽象接口 | 实际代码中的实现 | 说明 |
|-----------------|-----------------|------|
| `check_word_exists_api()` | `mdx_reader.lookup(word)` 返回 `(result, exact)`<br>外层包装 `is_valid_word(word) -> bool` | `main.py` 第246-251行：判断单词是否在 MDX 词典中存在 |
| `get_word_frequency_api()` | `mdx_reader.get_rank(word)` 返回 `int\|None`<br>HTTP接口: `GET /api/rank?word=xxx` | `main.py` 第226-234行：从词典的 `<span class="rank">` 标签提取 COCA 词频排名（数值越小越常用） |
| `BNC_COCA_lists.csv` | ❌ 当前代码中**未使用**该文件 | 规划中建议的外部词频表，尚未接入 |

### 实际使用示例（当前代码）

**判断单词是否存在：**
```python
result, exact = mdx_reader.lookup(word)
if exact:
    # 单词存在
```

**获取词频：**
```python
rank_val = mdx_reader.get_rank(word)
# rank_val 为 int 表示词频排名，None 表示无排名数据
```

### 建议

对于阶段4的"自适应闭环流程"，需要将上述方法封装成两个独立、可插拔的函数（如 `word_exists(word) -> bool` 和 `get_frequency(word) -> int|None`），以便在推演逻辑中干净地调用。当前代码中 `is_valid_word` 是 `affix_loader.analyze()` 的回调参数，不是全局函数，需要重构。

## 五、开发实现要点（对后续写代码的指导）

### 5.1 数据清洗（地基工程）
由于 Anki 原生数据脏、存在重复或 `nv/vn` 等无效占位符，在"阶段1"读取 CSV 时必须先做：
- **去重**：去除重复条目
- **标准化**：统一单词格式（小写、去除空格、特殊字符）
- **格式校验**：验证切分格式（`prefix.stem.suffix`）的正确性
- **异常处理**：处理空字段、无效切分、乱码等

### 5.2 API 接口解耦
编写 Python 核心逻辑时，要将 **存在性校验** 和 **词频获取** 设计成两个标准的可插拔函数接口，便于独立维护和测试。

### 5.3 推荐技术栈
- **Python**（生态最成熟）
- **文本处理**：`textdistance` 或 `Levenshtein` 库（编辑距离）
- **数据存储**：`SQLite` 或直接读写 `JSON/CSV`

### 5.4 实施顺序建议
1. **第一步**：数据清洗 + 知识沉淀（读 CSV，构建前后缀矩阵 + 辅音骨架库）
2. **第二步**：新词拆解（最长匹配剥离前后缀）
3. **第三步**：模糊推演（换头/改尾/动内脏三条路径）
4. **第四步**：API 闭环校验 + 评分排序 + 分级输出
